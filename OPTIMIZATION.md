# context-pruner v2.2 优化方案：三层过期判定

> **⚠️ 已被 v3.0 取代 — v2.2 逻辑已整合进 v3（Layer B + Layer C），且 v3 新增 Layer A 健康门禁。详见 `DESIGN-v3.md`。**
>
> 日期：2026-07-01
> 版本：v2.1.1 → v2.2
>
> **核心问题：** 当前裁剪只认"有没有引用路径"这一种依据，无法判断消息是否属于已结束的对话线程。
> 结果：assistant 基线 1.5 < SCORE_KEEP 2.0 后 → 中间推理线可以进入候选池了，但新问题是被裁区域可能包含仍然相关的内容。
>
> **设计目标：** 从"一刀切保活"转向"准确找出过时内容"，保留率目标 80-90%。

---

## 问题再确认

当前 2.1.1：
- assistant 基线 1.5 → 死锁解除，保留率 ~50%
- 高频率每轮都裁剪（~15s 间隔），但因为 skip cache 存在，消息不变化时跳过
- **真正问题：** 无法区分"当前活跃线程的消息"和"已结束线程的消息"
  - 话题 A 的 tool_output 被话题 B 的助理间接引用 → 不会裁（误保）
  - 话题 B 刚结束，助手消息还用了"上一轮"的引用路径 → 不该裁却裁了（误裁）
- 每轮重复裁到相同的消息是因为 skip cache 靠消息数+哈希判断，新消息加入后哈希改变 → 剪枝条件变，Skip 失效

### 需要区分的情况

| 场景 | 该不该裁 | 当前是否裁对 |
|:----|:--------:|:----------:|
| 话题结束了，消息无人再引用 | 裁 ✓ | 有时裁 | 
| 话题中间的分析推理，后续重复引用 | 不裁 | 可能被裁 |
| tool 输出被当前助理使用 | 不裁 | 可能被裁 |
| 很久以前的 tool 输出，同路径已被新数据取代 | 裁 ✓ | 不裁（误保） |
| 后端 config 定义（会话开头） | 不裁 | 当前有保护 |

## 三层方案

### 第一层：结构分区（~10 行）

| 分区 | 范围 | 策略 |
|:----|:----|:-----|
| **核心区** | msg 0-20（系统指令+首轮语境） | `PROTECTED_LEADING` → 永不入候选 |
| **活跃区** | 最近 3 user + 最近 5 assistant | 现有 `PROTECTED_USER` / `PROTECTED_ASST` → 全保留 |
| **老化区** | 中间全部 | 进入候选，按引用链判定 |

当前 `_prune()` 里的 protected 逻辑已经保护了 「系统+最近3条user+最近5条assistant」。但**核心区**（session 开头的函数签名、策略规则、项目结构）没有保护，评分系统不认"隐式引用"。

### 第二层：引用链追溯（~30 行）

核心思想：**一条消息只要还能通过引用链连回活跃区，就不算过时。**

算法：
1. 从活跃区（protected zone）获取`active_paths` / `active_refs` 作为**种子**
2. **从活跃区向回走**（反向遍历老化区消息）：
   - 检查当前消息的路径/引用是否出现在 `active_*` 中
   - 如果是 → 标记为 `chain_connected=True`，并将**它的**路径/引用也加入 `active_*`（链延伸）
   - 如果否 → 标记为 `chain_connected=False`（断链/孤立线程）
3. 评分时：`chain_connected`的消息才能获得引用加分

```
示例：
活跃区引用：/etc/nginx/nginx.conf
← 消息105（nginx.conf 路径出现）→ chain_connected ✓，加入 /etc/hosts
← 消息104（/etc/hosts 出现，但没有 nginx.conf）→ chain_connected ✓
← 消息103（只有 webpack.config.js，没有活跃区链路径）→ chain_connected ✗ → 断链
```

效果：孤立线程的消息 baseline 分数不变，但拿不到参考加分 → 更容易进入候选池。

### 第三层：数据覆盖检测（~15 行）

**问题：** 同一个文件读取两次的 tool 输出，旧的那条不再需要。

检测：
1. 在评分前扫描所有 tool 消息，建立 `路径 → 最后出现的索引` 映射
2. 评分时：如果一条 tool 消息的路径在更后面的位置有另一条 tool 消息也包含同一路径 → `superseded` 标记，减分

```
tool 89: cat /etc/config.yaml  → 500 行
... 10 轮后 ...
tool 120: cat /etc/config.yaml  → 新 500 行
            ↑ 路径相同，tool 89 已过时
```

## 限制与边界

- **引用链追溯只能识别显式路径/函数引用**，不能理解语义关联
- 如果一个话题完全依靠变量名（非文件路径、非大写常量）连接，可能判为断链 → 可接受的假阳性（被裁的消息即使误裁也在当前系统的误差范围内）
- 第三层只对同路径 tool 输出有效，不同路径的重复数据不会被覆盖检测
- 85-90% 保留率是合理预期，追求 >95% 会导致过时消息清理不足
