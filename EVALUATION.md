### 裁剪上限压力

| 指标 | 数值 |
|:----|:----:|
| 触及 `max_droppable` 上限 | **202/202 (100%)** |

## 三、代码质量审计

### 设计优点

- **非侵入式** — 中间件模式，不影响其他插件或核心逻辑
- **最佳错误处理** — 存档 I/O 全部 best-effort + 线程锁隔离
- **工具调用感知** — 显式处理 `assistant+tool` 块的原子性
- **清晰的分层架构** — register / middleware / core / archive 层次分明

### 已知问题

| 问题 | 严重度 | 说明 |
|:----|:-----:|:-----|
| `_CHECKPOINT_RE` 死代码 | 低 | 定义在第 25 行，import 了 `json` 和 `os` 仅用于存档，正则从未使用 |
| `SCORE_KEEP` 参数未校准 | 中 | 100% 触及上限，评分系统几乎不影响裁剪结果 |
| 短消息数下仍执行判定 | 低 | `if n <= PROTECTED_USER + PROTECTED_ASST + 50` 在 58 条以下跳过裁剪，但 58–200 区间仍执行全量评分 |
| 每次 LLM 请求都触发风险判断 | 低 | 尽管 skip cache 减少重复计算，但消息数量变化时仍重跑全量评分。 |

## 四、当前版本工作点总结

```
裁剪策略:  Always-On 评分裁剪  v2.0
保护层:    system(1) + user(3) + asst(5)
评分阈值:  SCORE_KEEP=2.0  ← 过低
实际限制:  MAX_PRUNED_PCT=60%  ← 满负荷
裁后保留:  35.9% (中位)
工作状态:  max_droppable 触顶运行
```

**核心矛盾：** 评分系统本应区分消息质量，但由于阈值过低，评分退化成了"非保护即候选"的二元分类。实际裁剪量完全由 60% 上限决定，评分系统未发挥应有作用。
