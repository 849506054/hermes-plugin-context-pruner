# context-pruner v3 设计方案：会话健康维护

> **⚠️ 已实现 — 2026-07-02 代码已部署至 `.hermes/plugins/context-pruner/`，待 Gateway 重启生效。**
>
> **日期：** 2026-07-02
> **版本：** v2.2 → v3.0
>
> **核心思路迁移：**
> ```
> 旧方向（已证伪）：
>   每轮瘦身 token 数 → 每次裁 55% → 缓存前缀全乱 → cache miss → 成本反升
>
> 新方向（本文）：
>   清除过期变质内容 → 保持会话健康 → 延迟压缩触发 → 缓存稳定命中 → 真实降本
> ```

---

## 一、问题总结：为什么 v2.x 反而增加了成本

### 1.1 运行数据回顾

来自 839 条 archive 记录的实证：

| 指标 | v2.1.1 + v2.2 |
|------|:-------------:|
| 平均每轮剪裁率 | 55.2% |
| 触及 MAX_PRUNED_PCT 上限比例 | 93.8% |
| 每次 LLM 调用触发 pruner | **是，100%** |
| 链连接有效识别率 | 238/839（28%）|
| 数据覆盖检测触发率 | 0/839（0%）|
| 单会话 pruner 调用峰值 | **238 次**（86 分钟内） |

### 1.2 缓存命中率下降的根因

**LLM 提供商缓存机制（DeepSeek prefix cache / Claude prompt caching）：**

```
缓存 key = 整个 prompt 的前 N 个 token 的精确匹配
```

- **无 pruner 时：** 会话从 [A, B, C, D] 增长到 [A, B, C, D, E]，前 4 条完全不变 → **cache HIT**
- **有 pruner 时：** 每次调用前 pruner 重新计算裁哪些 → [A, C, D, E] 或 [A, B, D, E] 或 [A, B, C, E]
  → **每次 prompt 前缀都不同 → 永远 cache MISS**

**关键洞察：** pruner 省掉了每次的 token 数量，但代价是每一次调用都是 cache miss 全价。省掉的 token 数远抵不上 cache miss 的损失。

### 1.3 更深层问题：v2.x 的触发策略是错误的

v2.x 的口号是 "Always-On"（每次 LLM 调用前都裁），这就注定了：

1. 每次调用都重新计算裁剪方案 → 输出不同的消息序列 → 缓存前缀被打散
2. 即使 skip cache 避免了重复计分，但**新消息加入 → 边界变化 → skip cache 失效 → 重新计分**
3. 因为评分阈值低（SCORE_KEEP=2.0），几乎所有非保护消息都是候选 → 每次裁法不同
4. 93.8% 触及上限 → 评分系统退化成了"非保护即裁"的二元过滤器，没有任何区分度

## 二、新设计哲学

### 2.1 核心定义

> **context-pruner 的目标不是「减少每次请求的 token 数」，而是「清除会话中已过期变质的内容，维持会话的健康性，从而延迟压缩触发，避免失真」。**

### 2.2 三个基本原则

| 原则 | 含义 | 设计影响 |
|:-----|:-----|:---------|
| **① 保护缓存前缀** | 消息序列的主体必须保持稳定 | 不改变消息排列顺序，不"填空"或重排。只从序列中移除确定已过时的消息节点 |
| **② 低触发频率** | 不在每次 LLM 调用时触发 | 只在"有可能触发压缩"时才介入（消息数 > 阈值或距离上次裁剪已足够远） |
| **③ 高精准度** | 只裁真正过时的内容，宁可误留不误裁 | 保留率目标 90-95%，assistant 误裁率 < 5% |

### 2.3 新旧指标对比

| 指标 | v2.x（旧） | v3（新） | 说明 |
|:-----|:--------:|:-------:|:-----|
| 触发频率 | 每次 LLM 调用 | 每 N 轮 / 达到阈值时 | N 取决于消息增长率 |
| 平均裁剪率 | 55% | **5-15%** | 少量精准清除 |
| 缓存命中率 | ~0% | **>85%**（前缀稳定） | 核心收益 |
| 会话寿命 | 90 分钟（238 调用后压缩） | **预计 4-8 小时** | 延迟压缩 |
| 评分区分度 | 失效（93.8% 打到上限） | 高（候选 < 20%） | 评分系统真正起作用 |

## 三、v3 架构设计

### 3.1 三层结构不变，但逻辑和目标全变

```
输入: messages[0..N]
  │
  ├── Layer A: 会话健康评估 ── 是否到达需要清理的阈值?
  │   ├── 消息数 > HEALTH_THRESHOLD_BASE (如 500)
  │   ├── 且 (距上次清理 > GROWTH_INTERVAL 或 新增消息数 > GROWTH_THRESHOLD)
  │   └── 不满足 → 跳过（保护缓存前缀，零开销）
  │
  ├── Layer B: 过期内容识别 ── 什么是"已过期变质"的?
  │   ├── ① 已完结的子话题（引用链完全断裂）
  │   ├── ② 被自我纠正的内容（后续消息明确修正了前文）
  │   ├── ③ 被覆盖的 tool 输出（同路径有更新的数据）
  │   └── ④ 已无用的中间推理（最终结论已被采纳，中间推理过程无引用）
  │
  └── Layer C: 序列稳定性保障 ── 裁剪后缓存还能命中
      ├── 只移除，不改序
      ├── 不移除 protected zone + active zone（与 v2.x 一致）
      └── 裁剪量上限从 60% 降为 20%
```

### 3.2 Layer A：会话健康评估（新增，最关键的变更）

这是与 v2.x 最核心的区别。不再是"每次 LLM 调用都裁"，而是：

```
每次 LLM 调用前：
  1. 消息数 < HEALTH_THRESHOLD_BASE (500)？→ 跳过，零开销
  2. 距上次清理 < GROWTH_INTERVAL (50 次调用)？→ 跳过
  3. 距上次清理后新增的消息数 < GROWTH_THRESHOLD (30 条)？→ 跳过
  4. 真正执行清理
```

**效果：**
- 会话从 60 条到 500 条之间，pruner **完全不触发** → 缓存前缀连续稳定 → cache HIT
- v2.x 下 238 次调用 = 238 次 pruner 触发
- v3 下 238 次调用（假设消息从 60 增长到 500）= **从第 500 条起每 50 轮清理一次 ≈ 只触发 2-3 次**

### 3.3 Layer B：过期内容识别（评分逻辑重写）

当前评分逻辑的问题：SCORE_KEEP=2.0 太低，几乎所有非保护消息都<2.0 → 候选池溢出。

新的评分设计：

```
基线分值（基础分 5.0，满分为 10 分的流畅标度）：

角色基线：
  system:     auto-protect（不参与评分）
  user:       6.0（高于 cutoff，绝大多数时间保留）
  assistant:  5.0（基线）
  tool:       4.0（基线低于 cutoff，但可以通过引用加分拉回）

加分项：
  +3.0  被后续消息显式引用（re: "如你所说"、"刚才那个错误"等）
  +2.0  引用链可追溯到 protected zone（当前链连接）
  +1.5  位于活跃话题的推理链中
  +1.0  包含代码、配置、或结构定义

减分项：
  -2.0  同路径有新 tool 输出覆盖（数据覆盖检测）
  -1.5  子话题已结束（引用链完全断裂超过 20 条消息）
  -1.0  已自我纠正（后续有明确纠正内容）

保留阈值：SCORE_KEEP = 4.5
  → 评分 ≥ 4.5 的消息保留
  → 评分 < 4.5 的消息进入候选池
```

**关键变化：**
1. 基线从 0-2 提升到 4-6，让评分系统有操作空间
2. SCORE_KEEP 从 2.0 提升到 4.5，让大部分消息保留
3. 加分项侧重"被引用"而非"有路径引用"（后者是主动引用，前者是被引用价值）

### 3.4 Layer C：序列稳定性保障

裁剪后：

```
裁剪前: [sys, msg1, msg2, msg3, ..., msgN-2, msgN-1, msgN]
裁剪后: [sys, msg1, msg2, msg3, ..., msgN-2, msgN-1, msgN]
         ↑ 前面部分完全不变                ↑ 最近活跃区完全不变
               ↑ 中间裁掉的节点仅从序列中移除，不改变剩余消息顺序
```

**保证：**
- 不改变任何消息的顺序
- 不"填空"或"合并"消息
- 裁剪量上限从 60% 降低到 **20%**（避免过度裁剪）

## 七、风险与边界

- **假阴性（该裁没裁）** — 宁可保留，等下一轮再裁。v3 的设计追求精准而非激进
- **会话过长仍会压缩** — v3 目标是延迟压缩，不是"永不压缩"。当会话达到 1000+ 条时仍然需要 checkpoint 机制
- **引用检测的局限性** — 只能识别显式引用模式（路径、函数名、错误名），不能理解语义关联。这是可接受的误差
- **最大风险：回退后缓存重建** — 如果 v3 效果不佳需要回退，缓存重建成本较高。建议先做 A/B 测试（只在一个 profile 上启用 v3）
